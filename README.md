# Soaring Heights Kennel Management System

Internal web application for managing dog boarding reservations, scheduling, billing, and PACFA compliance. Deployed on a homestead LAN (R630 host) for staff use and owner self-service via one-time portal links. Not internet-exposed.

## 2. Config Directory Inventory

All files are in `/config/` relative to the project root. They are mounted read-only into the backend container. Changes require root access and take effect on the next container restart.

### business.json
Business identity used in PDF receipts and email headers.
```json
{
  "name": "Soaring Heights Kennel",
  "address": "1234 Homestead Road, Strasburg, CO 80136",
  "phone": "303-555-0100",
  "email": "info@soaringheightskennel.com",
  "website": "https://soaringheightskennel.com",
  "logo_path": "/config/logo.png"
}
```

### kennels.json
Defines kennel types. On every startup, the backend auto-provisions individual kennel records (K-01, K-02, ...) for each type sequentially. Adding a type and restarting creates new kennels. Removing a type triggers conflict detection — deactivation is blocked if future reservations exist.
```json
{
  "kennel_types": [
    { "type": "Large", "sqft": 30, "max_size_class": "XL", "features": "interior and exterior space", "count": 8 },
    { "type": "Small", "sqft": 10, "max_size_class": "M",  "features": "interior only",              "count": 10 }
  ]
}
```

### pacfa.json
PACFA-required square footage by size class and stay duration. **Hard enforcement — no staff override permitted.** Formula: `(dog_length_inches + 6)² / 144 = required_sqft`.
```json
{
  "size_classes": {
    "XS": { "est_length_in": 12, "base_sqft": 2.25 },
    "S":  { "est_length_in": 18, "base_sqft": 4.00 },
    "M":  { "est_length_in": 24, "base_sqft": 6.25 },
    "L":  { "est_length_in": 30, "base_sqft": 9.00 },
    "XL": { "est_length_in": 36, "base_sqft": 12.25 }
  },
  "stay_multipliers": {
    "lt_5_days": 1.0,
    "5_to_30_days": 1.5,
    "31_to_180_days": 2.0,
    "181_plus_days": 3.0,
    "181_plus_days_with_daily_qualifying_activity": 2.0
  },
  "qualifying_activity_types": ["Nature Walk", "Play Yard"]
}
```

### phases.json
Day phase window start times in 24-hour `HH:MM` format. Phase is always derived from clock time; never stored directly.
```json
{
  "morning_start":   "06:00",
  "afternoon_start": "12:00",
  "evening_start":   "18:00",
  "night_start":     "00:00"
}
```

### ports.json
Container port assignments. Must match `docker-compose.yml` and `nginx/kennel.conf`.
```json
{
  "frontend_port": 9100,
  "backend_port":  9101
}
```

### pricing.json
Nightly boarding rates by dog size class and per-activity prices. These values are read-only from the UI. Changing them requires root access and a container restart.
```json
{
  "nightly_rates": { "XS": 25.00, "S": 30.00, "M": 35.00, "L": 40.00, "XL": 45.00 },
  "activity_prices": {
    "Nature Walk": 15.00, "Playtime": 10.00,
    "Medication Administration": 5.00, "Emergency Grooming": 30.00, "Play Yard": 12.00
  }
}
```

### smtp.json
SMTP credentials for automated receipt emails. `auto_email_receipt` in `system.json` must also be `true` for emails to send.
```json
{
  "server":       "smtp.example.com",
  "port":         587,
  "username":     "kennel@example.com",
  "password":     "CHANGE_ME",
  "from_address": "kennel@soaringheightskennel.com",
  "use_tls":      true
}
```

### system.json
Operational settings.
```json
{
  "auto_email_receipt":             false,
  "pickup_overdue_threshold_hours": 3,
  "portal_link_expiry_days":        7,
  "backup_retention_days":          30,
  "backup_path":                    "/mnt/nas/kennel_backups"
}
```
| Key | Effect |
|---|---|
| `auto_email_receipt` | If `true`, PDF receipt is emailed to owner on checkout |
| `pickup_overdue_threshold_hours` | Hours past scheduled pickup before overdue alert fires |
| `portal_link_expiry_days` | Days before a one-time owner portal link expires |
| `backup_retention_days` | Daily backups older than this are pruned automatically |
| `backup_path` | Path inside the container where backups are written (must be mounted NAS volume) |

## 3. First-Time Installation

### Prerequisites
- Docker + Docker Compose (v2)
- Node.js ≥ 18 and npm (on the R630 host, for the build step)
- Nginx running on the R630 host
- NAS volume accessible at a host path (set `NAS_BACKUP_PATH` below)

### Environment variables
Set these before running the build. Add to `/etc/environment` or the invoking shell:
```bash
export KMS_SECRET_KEY="$(openssl rand -hex 32)"   # Required — JWT signing key
export NAS_BACKUP_PATH="/mnt/nas"                  # Host path to NAS mount point
```
`KMS_SECRET_KEY` defaults to `CHANGE_ME_IN_PRODUCTION` if unset — **do not leave this in production**.

### Build and deploy
```bash
cd /home/home/claud_code_folder/code/kennel_mgmt_system
./build.sh
```
`build.sh` performs four steps in sequence:
1. `npm install && npm run build` in `frontend/`
2. `docker compose build`
3. `docker compose up -d`
4. `nginx -s reload`

### First-run verification
After `build.sh` completes:
```bash
# Confirm containers are up
docker compose ps

# Confirm backend seeded correctly (should list 5 activity types)
docker compose exec backend python manage.py list-users
# → "No users found." on a fresh install (expected)

# Health check
curl -s http://127.0.0.1:9101/api/health
# → {"status": "ok"}

# Create the first staff account
docker compose exec backend python manage.py add-user admin yourpassword
```
The database, kennel records (K-01 through K-18 from default kennels.json), and activity type seed data are all created automatically on the first backend startup.

## 5. Staff Account Management

All staff account operations run inside the backend container via `manage.py`. There is no UI for user management.

```bash
# Add a new staff user
docker compose exec backend python manage.py add-user <username> <password>

# Change a staff user's password
docker compose exec backend python manage.py set-password <username> <new_password>

# Deactivate a staff user (blocks login; does not delete)
docker compose exec backend python manage.py deactivate-user <username>

# List all staff users with status and creation date
docker compose exec backend python manage.py list-users
```

Passwords are bcrypt-hashed. Deactivated users cannot log in but their audit trail records are preserved. There is no reactivation command; to reactivate, connect to the database directly and set `active = 1` in the `staff_users` table.

## 6. Database Migration

### How migrations are applied
On every container startup, `main.py` scans `backend/migrations/*.sql` in filename order and applies any file not yet recorded in the `schema_migrations` table. Each applied filename is inserted into `schema_migrations` so it is never re-applied.

The database file lives at `/data/kennel.db` inside the `kennel_data` Docker volume.

### Adding a new migration
1. Create `backend/migrations/002_your_description.sql` (increment the numeric prefix).
2. Write idempotent DDL — use `CREATE TABLE IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`, etc.
3. Restart the backend container:
   ```bash
   docker compose restart backend
   ```
4. Confirm the migration was applied:
   ```bash
   docker compose exec backend python -c "
   import asyncio, os
   os.environ.setdefault('DATABASE_URL', 'sqlite+aiosqlite:///data/kennel.db')
   os.environ.setdefault('CONFIG_DIR', '/config')
   async def check():
       from app.database import async_engine
       from sqlalchemy import text
       async with async_engine.connect() as c:
           rows = await c.execute(text('SELECT filename, applied_at FROM schema_migrations ORDER BY applied_at'))
           for r in rows: print(r)
   asyncio.run(check())
   "
   ```

### Manual migration round-trip test
```bash
./scripts/qa.sh --skip-e2e --skip-bandit
```
The `6/6 Schema migration round-trip` step applies migrations to a temp DB, drops all tables, and re-applies — verifying idempotency.

## 7. NAS Backup

### How it works
A cron job inside the backend container runs `scripts/backup.py` daily at **1:15 AM**. It copies `/data/kennel.db` to the NAS mount at the path defined by `backup_path` in `system.json`, naming each file `kennel_db_YYYY-MM-DD.sqlite`. Backups older than `backup_retention_days` days are pruned automatically.

### NAS mount setup
The host NAS path is passed to the container via the `NAS_BACKUP_PATH` environment variable in `docker-compose.yml`:
```yaml
volumes:
  - ${NAS_BACKUP_PATH:-/tmp/kennel_backups}:/mnt/nas
```
Ensure the NAS is mounted on the R630 host at `NAS_BACKUP_PATH` before starting containers. If the path is missing or unmounted, backups will silently write to `/tmp/kennel_backups` (non-persistent).

### Verify backup cron is scheduled
```bash
docker compose exec backend crontab -l
# Should show: 15 1 * * * python /app/scripts/backup.py
```

### Run a manual backup
```bash
docker compose exec backend python /app/scripts/backup.py
# Output: Backup written: /mnt/nas/kennel_backups/kennel_db_YYYY-MM-DD.sqlite
```

### Verify retention pruning
```bash
docker compose exec backend ls -lh /mnt/nas/kennel_backups/
```

## 8. Key Business Rules

### Phase Windows
The day is divided into four named phases based on clock time. Phase is always computed from the current time; it is never stored.

| Phase | Window |
|---|---|
| Night | 00:00 – 05:59 |
| Morning | 06:00 – 11:59 |
| Afternoon | 12:00 – 17:59 |
| Evening | 18:00 – 23:59 |

Start times are configurable in `config/phases.json`. Phase boundary minutes are inclusive on the start (06:00:00 = Morning, 11:59:59 = Morning, 12:00:00 = Afternoon).

### Post-Checkout Hold Rule
After checkout, the kennel is placed on hold for the remainder of the checkout phase. The hold phase and the next available phase depend on which phase checkout occurred in:

| Checkout Phase | Hold Through | Next Available Phase |
|---|---|---|
| Morning | End of Morning | Afternoon |
| Afternoon | End of Afternoon | Evening |
| Evening | End of Evening | Next Morning |
| Night | End of Night | Morning |

Evening checkout creates a hold that extends to the following Morning — the Evening-to-Night edge case.

### PACFA Enforcement
All reservations are validated against Colorado PACFA space requirements. **This is a hard enforcement rule with no staff override.** Required square footage is calculated as:

```
required_sqft = (dog_length_inches + 6)² / 144
```

A stay-duration multiplier is applied:

| Stay Duration | Multiplier |
|---|---|
| < 5 days | 1.0× |
| 5 – 30 days | 1.5× |
| 31 – 180 days | 2.0× |
| 181+ days | 3.0× (or 2.0× with daily qualifying activity) |

**181+ day exception:** If a Nature Walk or Play Yard activity is scheduled and confirmed for every day of the stay, the 2.0× multiplier applies. If any day is missing a confirmed qualifying activity, the full 3.0× applies for the entire stay. The system alerts staff each Morning phase when a 181+ day stay dog has no confirmed qualifying activity for that day.

**Multi-dog same-kennel:** When multiple dogs share a kennel, their required sqft values are summed and compared against the kennel's total sqft. The longest individual stay drives the multiplier for each dog independently.

### 14-Day Billing Cycle
For stays exceeding 14 days, billing is split into 14-day cycles (Cycle 1, Cycle 2, etc.). Each cycle generates a separate Bill record. An alert fires to staff 24 hours before a cycle boundary to prompt collection.

### Phase Conflict Override
A reservation whose drop-off or pickup phase conflicts with an occupied phase can be overridden by staff with a checkbox. The override is logged to `reservation.override_log` with the staff username, timestamp, and conflict description.

## 9. API Base URL and Conventions

### Base path
All API endpoints are prefixed with `/api/`:
```
http://kennel.soaringheights.local/api/
```
Or directly on the R630:
```
http://127.0.0.1:9101/api/
```

### Authentication
1. Obtain a token (valid 8 hours):
   ```bash
   curl -s -X POST http://127.0.0.1:9101/api/auth/login \
     -d "username=admin&password=yourpassword"
   # → {"access_token": "<JWT>", "token_type": "bearer"}
   ```
2. Include on all subsequent requests:
   ```
   Authorization: Bearer <JWT>
   ```
   Unauthenticated requests return `401 Unauthorized`.

### Exceptions to auth requirement
- `POST /api/auth/login` — no token required
- `GET /api/portal/{token}` and all `/api/portal/` sub-routes — one-time token in URL path

### Error response structure
All errors return JSON with `detail`:
```json
{ "detail": "Human-readable error message" }
```
Validation errors (422) include a `detail` array:
```json
{
  "detail": [
    { "loc": ["body", "field_name"], "msg": "field required", "type": "value_error.missing" }
  ]
}
```

### Interactive API docs
Swagger UI is available at `http://kennel.soaringheights.local/docs` and allows browsing and testing all endpoints directly. All endpoints are documented with docstrings.

## 10. Common Troubleshooting

### Container not starting

**Symptom:** `docker compose ps` shows a container in `Restarting` state.

```bash
docker compose logs backend   # or: logs frontend
```

Common causes:
- `KMS_SECRET_KEY` not set — defaults to `CHANGE_ME_IN_PRODUCTION`, which works but logs a warning. Not a crash cause.
- `CONFIG_DIR` missing or config files malformed — backend crashes on startup if any JSON config file is missing or invalid. Validate JSON: `python -m json.tool config/system.json`
- Database volume permissions — verify `kennel_data` volume is writable: `docker compose exec backend ls -la /data/`
- Port conflict — another process may be using 9100 or 9101: `ss -tlnp | grep -E '9100|9101'`

### Nginx proxy errors (502 Bad Gateway)

The containers are bound to `127.0.0.1` only. Nginx must be on the same host.

```bash
# Confirm containers are running and listening
ss -tlnp | grep -E '9100|9101'

# Test backend directly (bypassing Nginx)
curl -s http://127.0.0.1:9101/api/health

# Check Nginx error log
sudo tail -50 /var/log/nginx/error.log

# Revalidate config after any nginx.conf change
sudo nginx -t && sudo nginx -s reload
```

### Migration failures

**Symptom:** Backend container starts and immediately exits; logs show a migration error.

```bash
docker compose logs backend | grep -i migration
```

Common causes:
- Syntax error in a new `.sql` file — test against a temp DB first:
  ```bash
  sqlite3 /tmp/test.db < backend/migrations/002_your_migration.sql
  ```
- Duplicate `CREATE TABLE` without `IF NOT EXISTS` — always use `IF NOT EXISTS` in migration DDL.
- `schema_migrations` table was manually deleted — the migration runner will re-apply all files. If the schema already exists, this causes constraint errors. Restore from backup or manually re-insert the applied filenames.

### Backup not running

```bash
# Verify cron is present in the container
docker compose exec backend crontab -l

# Check if NAS is actually mounted
docker compose exec backend ls /mnt/nas/

# Run backup manually and check output
docker compose exec backend python /app/scripts/backup.py

# Check container logs for cron output (cron writes to stdout/stderr)
docker compose logs --since 24h backend | grep -i backup
```

If `/mnt/nas/` is empty or shows the host's `/tmp/kennel_backups`, the `NAS_BACKUP_PATH` environment variable is not set or the NAS volume is not mounted on the host. Mount the NAS first, then restart the container:
```bash
sudo mount -t nfs <nas-ip>:/share /mnt/nas
export NAS_BACKUP_PATH=/mnt/nas
docker compose up -d
```
